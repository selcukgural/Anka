## How to Interpret These Results

| Metric | What it measures |
|---|---|
| **Req/sec** | Peak throughput — how many complete HTTP/1.1 request–response cycles Anka finishes per second. Higher is better. |
| **Avg Latency** | Mean time from request sent to first byte of response received. Lower is better. |
| **P90 / P99** | 90th / 99th percentile latency. P99 captures tail latency — the worst 1 in 100 requests. |
| **Transfer/s** | Network throughput in MB/s (headers + body). Useful for comparing response-size scenarios. |
| **Errors** | Non-2xx responses or connection failures as a percentage of total requests sent. |

### Scaling pattern

Anka uses a **fire-and-forget task per connection** model: the accept loop never blocks.
Throughput therefore scales with connection count until the CPU or OS scheduler saturates.
The "sweet spot" is typically where Req/sec plateaus — adding more connections after that
increases latency without improving throughput.

### Key design properties visible in the numbers

- **Zero heap allocation per request** — `ArrayPool<byte>` + `HttpRequestPool` CAS slot keeps
  GC pressure flat even at high concurrency.
- **Single `SendAsync` for small bodies** (≤ 4 KB) — headers and body are written into one
  rented buffer, producing a single syscall. The *Large Response* scenario triggers the
  two-`SendAsync` path.
- **HTTP/1.1 keep-alive** — wrk reuses connections; all measured latency is pure
  server-processing time, not TCP handshake overhead.
